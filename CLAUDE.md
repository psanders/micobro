# Micobro — Agent Guide

Offline-first Android app (Expo/React Native) for independent, informal lenders
(prestamistas) in the Dominican Republic to track customers, loans, and payments —
no backend server; local SQLite is the source of truth on-device, synced to a
Google Sheet each lender owns.

## How work is organized

- **Coding conventions (the HOW)** live in this file. They apply to every change.
- **Product behavior (the WHAT)** lives in OpenSpec specs under `openspec/specs/`,
  changed through proposals in `openspec/changes/`. Use `/openspec:propose`,
  `/openspec:apply`, `/openspec:archive`. Specs describe observable, testable
  behavior — not coding style.
- **Shipping a change (the LOOP)** drives one change from design to archive with
  `/ps:ship <change>`: design → spec reconcile → build → tests → sync → archive,
  resumable via a per-change checkpoint.

## Repository layout

Single package — mobile is the only surface, so there's no monorepo/`common` split.

- `app/` — Expo Router screens and layouts.
- `components/` — shared UI components (and their `*.stories.tsx` for Storybook).
- `lib/db/` — drizzle-orm schema + client for the on-device SQLite database
  (`expo-sqlite`). `lib/db/migrations/` is generated by `npm run db:generate` —
  never hand-edit it.
- `lib/sync/` — Google Sheets integration: OAuth (`googleAuth.ts`, Google Sign-In
  per lender via PKCE — no service-account secret ships in the app), a REST
  client over Sheets API v4 (`sheetsClient.ts`, plain `fetch` — not the
  `googleapis` SDK, which assumes a Node runtime), and the push queue
  (`push.ts`) that replays `pending_mutations` rows. Pull/two-way sync and
  conflict resolution are not built yet — see `openspec/` for proposing that.
- `lib/<domain>/` (e.g. `lib/customers/`) — one `create<Name>.ts` file per
  business operation, its Zod schema, and a barrel `index.ts`. `lib/customers/`
  is the reference implementation.
- `lib/errors/`, `lib/utils/` — `ValidationError` and
  `withErrorHandlingAndValidation`, the spine of the validated-function pattern.
- `lib/logger.ts` — console-based logger. **Do not add `@fonoster/logger`** —
  it pulls in `winston`/`fluent-logger`, both Node-only and unusable under Hermes.
- `__tests__/` — one Jest test file per function.
- `.maestro/` — E2E smoke flows (machine-level Maestro CLI, not an npm dep).

## Coding conventions

### Validated functions (preferred pattern for business logic)

Every unit of business logic is a factory that injects dependencies (the db
handle, sync client, …) and wraps an inner `fn` with
`withErrorHandlingAndValidation(fn, schema)`, so invalid input throws a
structured `ValidationError` before anything runs, and tests inject a stubbed
`db` with no real SQLite. See `lib/customers/createCustomer.ts` for the
worked example. Full guide: `/ps:create-validated-function`.

### General

- TypeScript strict; no `any` (ESLint enforces `@typescript-eslint/no-explicit-any`).
- **No `.js` extension on relative imports.** This is a Metro-bundled RN app,
  not a Node ESM service — Metro resolves `.ts`/`.tsx` directly.
- Zod schemas import from `zod/v4`.
- **Jest, not mocha** — Expo/React Native requires the `jest-expo` preset for
  RN module/JSX transforms; mocha has no equivalent. Use `jest.fn()` for
  stubs, not sinon.
- **Prisma is not viable on-device** — its query engine needs a native binary
  or WASM, and Hermes supports neither. Local persistence is `drizzle-orm` on
  `expo-sqlite`.
- All user-facing text is Spanish (the target users are Dominican lenders);
  there's no i18n layer yet — hardcoded Spanish strings are intentional for now.

## Commits

Use **Conventional Commits** (`type(scope): subject`, e.g. `feat(customers): add phone validation`).
Not currently enforced by a commit-msg hook — `husky` only runs `lint-staged` on `pre-commit`.
